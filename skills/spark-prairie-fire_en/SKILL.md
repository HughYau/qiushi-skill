# A Single Spark Can Start a Prairie Fire

> "Here we can use the Chinese saying: 'A single spark can start a prairie fire.' That is to say, although the force is small at present, its development will be very rapid. In China's environment, it not only possesses the possibility of development but indeed the inevitability of development."
> — Mao Zedong, "A Single Spark Can Start a Prairie Fire"《星星之火，可以燎原》(1930)

## Core Principle

**Starting small does not mean having no future. The key is to establish a solid "base area" (foundation) rather than roaming without purpose (scattered action). Under the right conditions, a single spark can absolutely develop into a prairie fire.**

> See original-texts.md for detailed source references.

## When NOT to Use

**This skill is unnecessary in the following situations:**
- The project already has a solid foundation and is in the expansion phase, not starting from zero
- The user has already identified the entry point and only needs execution
- The task involves modifying an existing system, not building new — the existing codebase is the "base area"
- Ample resources are already available; the cautious bootstrap strategy of "a single spark" is unnecessary

## When to Use

Invoke this skill when:

- Starting a new project or task from scratch
- Resources are extremely limited and the grand objective seems impossible
- Facing an ambitious goal with no idea where to begin
- Doing many scattered things but accumulating nothing
- Needing to validate the feasibility of an idea with minimal investment
- Feeling that your forces are small and the prospects uncertain

## Method Workflow

### Step 1: Assess Objective Conditions

> "In China's environment, it not only possesses the possibility of development but indeed the inevitability of development." — "A Single Spark Can Start a Prairie Fire"《星星之火，可以燎原》

Not every "single spark" can start a prairie fire — the key lies in the conditions:
- Does a genuine need exist? (Is there "dry kindling"?)
- Do the objective conditions permit development? (Is the environment supportive?)
- Is my direction correct? (Direction matters more than speed)
- Is there a foundation to rely on? (However small)

**If conditions are not met:** First create the conditions (invoke `investigation-first` for investigation and research), or adjust direction.
**If conditions are met:** Have firm confidence and start small.

### Step 2: Select the Base Area

> Establish base areas rather than practicing roving banditry — "A Single Spark Can Start a Prairie Fire"《星星之火，可以燎原》

The "base area" is the first and most solid foundation you establish. Selection criteria:
- **Choose the entry point where you have the strongest advantage** — just as selecting a base area means choosing ground "favorable to us"
- **This entry point must be able to hold** — not something done once and discarded, but something that can develop continuously
- **This entry point must be expandable** — from here, you can extend outward to a larger scope

**Applied to specific scenarios:**
- New project → First build the most core, minimal viable functional module
- New domain → First establish deep understanding in one sub-domain
- New capability → First master one foundational skill as a starting point

### Step 3: Put Down Roots — Reject Roving Banditry

**Symptoms of roving banditry:**
- Doing a bit here and a bit there, accumulating nothing
- Constantly switching directions, barely scratching the surface of each
- Pursuing the appearance of "doing many things" while lacking real depth
- No reusable results; starting from zero every time

**Base area mindset:**
- Dig deep roots at a single point
- Each step's results build the foundation for the next
- Build reusable infrastructure and knowledge
- Have a clear expansion path

### Step 4: Accumulate Small Victories into a Great Victory

> Force grows from small to large, from "a single spark" to "a prairie fire" — "A Single Spark Can Start a Prairie Fire"《星星之火，可以燎原》

Expand systematically from the base area foundation:
- Expand one small step at a time, ensuring each step holds firm
- Consolidate each small achievement before extending outward
- Maintain patience — the accumulation of quantitative change into qualitative change takes time
- Record and summarize the experience of each expansion

### Step 5: Reject Two Kinds of Errors

**Pessimism (capitulation):**
> "Although the force is small at present, its development will be very rapid."

Symptom: "I have so few resources; I can't accomplish anything."
Correction: When conditions are right, a small force can absolutely grow powerful. Do not be intimidated by the current state of "smallness."

**Adventurism (roving bandit mentality):**
Symptom: "Never mind the plan — just do something. Attack in all directions."
Correction: Action without a base area is blind. First establish a foundation; develop with a plan.

## Four Elements of Base Area Construction

1. **Core functionality** — The most basic functions of the base area must be reliable
2. **Reusability** — What is built can be reused by subsequent work
3. **Extensibility** — There is a clear path to expand outward from here
4. **Solidity** — Verified through practice (`practice-cognition`), confirmed to be stable

## Common Mistakes

| Mistake | Mao Zedong's Critique | Correct Approach |
|---------|----------------------|-----------------|
| Feeling that small forces are meaningless | "A single spark can start a prairie fire" | Assess conditions; start small |
| Attacking everywhere without focus | Oppose "roving banditry" | Select a base area; dig deep roots |
| Rushing to expand before the base area is stable | The base area must be consolidated | Stabilize first, then expand |
| Acting blindly without assessing conditions | Check whether "dry kindling" is present | Assess objective conditions first |
| Barely scratching the surface, constantly switching | Base area mindset demands depth | Persist in depth on one direction |

## Operating Procedure

When this skill is triggered, **output a base area selection analysis and development roadmap**:

1. **Assess objective conditions** (before any action):
   ```
   Does a real need exist: [Yes/No/Uncertain] — Basis: ...
   Existing foundation to rely on: ... (however small)
   Main obstacles: ...
   Condition assessment: [Conditions for development exist / Conditions insufficient, must create conditions first]
   ```
   Conditions insufficient → Invoke `investigation-first` first; do not rush to act

2. **Select the base area** (one specific entry point):
   ```
   Base area selection: [What specifically]
   Selection rationale:
   - My advantage here: ...
   - Evidence this can hold firm: ...
   - Expansion path from here: ...
   ```

3. **Roving banditry check** (must pass before each expansion):
   - [ ] Has the previous entry point been verified as stable? (Do not expand if unstable)
   - [ ] Is this expansion a natural extension from the previous base area? (Leapfrog expansion is roving banditry)
   - [ ] Is the new addition reusable? (One-off work does not count as base area construction)

4. **Development roadmap** (3 steps; do not plan too far ahead):
   ```
   Step 1: [Specific action] → Success signal: ...
   Step 2: [Specific action] → Success signal: ... (after Step 1 is stable)
   Step 3: [Specific action] → Success signal: ... (after Step 2 is stable)
   ```
